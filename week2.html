Week 2: Django URLs, Views, and Templates

Building dynamic web pages with Django’s request-response cycle
1

URL Routing with path() and re_path()

Django uses a URL dispatcher to map URLs to views. This is defined in urls.py files.

path() – Simple & Readable

Uses path converters for common patterns (e.g., int, str, slug).

# myapp/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path('', views.home, name='home'),
    path('about/', views.about, name='about'),
    path('post//', views.post_detail, name='post_detail'),
    path('user//', views.profile, name='profile'),
]

re_path() – For Complex Patterns

Uses regular expressions when path converters aren’t sufficient.

from django.urls import re_path

urlpatterns = [
    re_path(r'^archive/(?P[0-9]{4})/$', views.year_archive),
]

Best Practice:

  • Prefer path() over re_path() for readability
  • Always use the name parameter for reverse URL lookups
  • Include app URLs in the main project using include()

Function-Based Views (FBVs)

Views are Python functions that receive an HttpRequest and return an HttpResponse.

Basic FBV Structure:

# views.py
from django.http import HttpResponse
from django.shortcuts import render

def home(request):
    return HttpResponse("Welcome to the homepage!")

def about(request):
    context = {'title': 'About Us', 'team_size': 12}
    return render(request, 'about.html', context)

View Responsibilities:

  • Process incoming request data (GET/POST parameters)
  • Interact with models (query database)
  • Pass data to templates via context dictionary
  • Return rendered HTML or redirect

Handling Requests:

def contact(request):
    if request.method == 'POST':
        name = request.POST.get('name')
        email = request.POST.get('email')
        # Process form data
        return HttpResponse("Thank you!")
    else:
        return render(request, 'contact.html')

Django Template Language (DTL): Variables & Tags

DTL allows embedding dynamic content into HTML using a simple, secure syntax.

Variables:

Render data passed from views using double curly braces.

<!-- In template: about.html -->
<h1>{{ title }}</h1>
<p>Our team has {{ team_size }} members.</p>

Built-in Tags:

  • {% if condition %} ... {% endif %}
  • {% for item in items %} ... {% endfor %}
  • {% url 'view_name' %} – Reverse URL lookup
  • {% csrf_token %} – Security for forms
<ul>
{% for post in posts %}
    <li>{{ post.title }} – {{ post.created_at|date:"M d, Y" }}</li>
{% empty %}
    <li>No posts yet.</li>
{% endfor %}
</ul>

DTL Filters and Security

Filters modify variables in templates. They are applied using the pipe (|) symbol.

Common Built-in Filters:

{{ name|upper }}                → Converts to uppercase
{{ bio|truncatewords:15 }}     → Truncates after 15 words
{{ price|floatformat:2 }}      → Formats as 19.90
{{ date|date:"F j, Y" }}       → Formats date as "January 1, 2024"
{{ content|linebreaks }}       → Converts \n to <p> and <br>

Automatic Escaping:

Django automatically escapes HTML in variables to prevent XSS attacks.

<!-- If user_input = "<script>alert('xss')</script>" -->
{{ user_input }}  
<!-- Renders as harmless text, not executable script -->

To disable escaping (only when safe), use the |safe filter:

{{ trusted_html|safe }}

Template Inheritance

Template inheritance allows you to define a base template and extend it in child templates—reducing duplication.

Base Template (base.html):

<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}My Site{% endblock %}</title>
    <link rel="stylesheet" href="{% static 'css/style.css' %}">
</head>
<body>
    <header>My Website</header>
    <main>
        {% block content %}{% endblock %}
    </main>
    <footer>© 2024</footer>
</body>
</html>

Child Template (home.html):

{% extends "base.html" %}

{% block title %}Home – My Site{% endblock %}

{% block content %}
    <h1>Welcome!</h1>
    <p>This is the homepage.</p>
{% endblock %}

Key Benefits:

  • DRY (Don’t Repeat Yourself) principle
  • Easier maintenance – change layout in one place
  • Consistent site structure

Static Files (CSS, JS, Images)

Static files are assets like CSS, JavaScript, and images served alongside your HTML.

Project Structure:

  • myapp/static/myapp/style.css
  • myapp/static/myapp/images/logo.png

Using Static Files in Templates:

{% load static %}
<!DOCTYPE html>
<html>
<head>
    <link rel="stylesheet" href="{% static 'myapp/style.css' %}">
</head>
<body>
    <img src="{% static 'myapp/images/logo.png' %}" alt="Logo">
    <script src="{% static 'myapp/script.js' %}"></script>
</body>
</html>

Settings:

  • Django finds static files automatically if STATIC_URL = '/static/'
  • During development, runserver serves static files automatically
  • In production, configure web server (e.g., Nginx) or use collectstatic

Lab: Create a Multi-Page Static Site

Requirements:

  • Create pages: Home, About, Contact
  • Use a shared base template
  • Include static CSS file
  • Use DTL variables and tags

Implementation:

# urls.py (app level)
from django.urls import path
from . import views

urlpatterns = [
    path('', views.home, name='home'),
    path('about/', views.about, name='about'),
    path('contact/', views.contact, name='contact'),
]
# views.py
from django.shortcuts import render

def home(request):
    return render(request, 'pages/home.html')

def about(request):
    return render(request, 'pages/about.html', {'team': ['Alice', 'Bob', 'Charlie']})

def contact(request):
    return render(request, 'pages/contact.html')

Lab Steps:

  1. Create a new app: python manage.py startapp pages
  2. Add 'pages' to INSTALLED_APPS
  3. Create templates/base.html and page templates in templates/pages/
  4. Create static/pages/style.css
  5. Define views and URL patterns
  6. Test all pages in browser
