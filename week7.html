What Are Class-Based Views?

Class-Based Views (CBVs) are views implemented as Python classes instead of functions.

  • Built on object-oriented principles (inheritance, mixins, etc.)
  • Part of Django’s django.views.generic
  • Handle HTTP methods (GET, POST, etc.) via class methods
  • Provide reusable patterns for common web development tasks

Key Insight:

  • CBVs promote code reuse and reduce boilerplate
  • They follow the "convention over configuration" principle
3

Advantages of CBVs over FBVs

  • Code Reusability: Inheritance and mixins reduce duplication
  • Organization: Related logic grouped in one class
  • Built-in Functionality: Generic views handle common patterns (CRUD)
  • Maintainability: Easier to extend and override specific behavior
  • Consistency: Standardized structure across projects

Side-by-Side Comparison:

# Function-Based View (FBV)
def post_list(request):
    posts = Post.objects.all()
    return render(request, 'blog/post_list.html', {'posts': posts})

# Class-Based View (CBV)
class PostListView(ListView):
    model = Post
    template_name = 'blog/post_list.html'
    context_object_name = 'posts'
4

Common Generic CBVs

The "Big Five" generic views for CRUD operations:

  • ListView – Display a list of objects
    model, template_name, context_object_name
  • DetailView – Show details of a single object
    model, template_name, slug_field
  • CreateView – Handle object creation (form + POST)
    model, fields or form_class, success_url
  • UpdateView – Edit an existing object
    model, fields, template_name_suffix='_form'
  • DeleteView – Confirm and delete an object
    model, success_url, template_name_suffix='_confirm_delete'
5

How Generic CBVs Work

Generic CBVs automatically handle common web patterns:

  • GET/POST request routing to appropriate methods
  • Form rendering and validation
  • Success redirects after form submission
  • Object lookups (via pk or slug)
  • Convention over configuration (e.g., default template names like blog/post_list.html)

Behind the Scenes:

  • ListView calls get_queryset() → renders template with context
  • CreateView handles GET (empty form) and POST (validation + save)
  • DeleteView shows confirmation template on GET, deletes on POST
6

Customizing CBVs

Override methods or attributes to tailor generic views:

  • get_queryset() – Customize which objects are displayed
  • get_context_data() – Add extra context to the template
  • form_valid() – Add logic after successful form submission
  • get_success_url() – Control redirect after success
  • template_name / context_object_name – Customize template behavior

Customization Examples:

def get_queryset(self):
    return Post.objects.filter(published=True)

def get_context_data(self, **kwargs):
    context = super().get_context_data(**kwargs)
    context['page_title'] = 'Published Posts'
    return context

def form_valid(self, form):
    form.instance.author = self.request.user
    return super().form_valid(form)

Example: Refactoring Blog Views

Before (FBV):

def post_list(request):
    posts = Post.objects.all()
    return render(request, 'blog/post_list.html', {'posts': posts})

def post_detail(request, pk):
    post = get_object_or_404(Post, pk=pk)
    return render(request, 'blog/post_detail.html', {'post': post})

After (CBV):

class PostListView(ListView):
    model = Post
    template_name = 'blog/post_list.html'
    context_object_name = 'posts'

class PostDetailView(DetailView):
    model = Post
    template_name = 'blog/post_detail.html'

Refactoring Tip:

  • Start with simple views (ListView, DetailView) before tackling forms
  • Templates often require no changes when switching to CBVs

Lab: Refactor Blog Views to CBVs

Task:

Refactor your existing blog app views (list, detail, create, update, delete) from FBVs to CBVs.

Steps:

  • Replace post_list with ListView
  • Replace post_detail with DetailView
  • Replace create/update forms with CreateView and UpdateView
  • Implement DeleteView with confirmation
  • Customize as needed (e.g., restrict editing to author)
  • Bonus: Override form_valid to auto-set author on create

Solution Snippet:

class PostCreateView(LoginRequiredMixin, CreateView):
    model = Post
    fields = ['title', 'content']
    success_url = reverse_lazy('post_list')
    
    def form_valid(self, form):
        form.instance.author = self.request.user
        return super().form_valid(form)

Lab Checklist:

  1. Update views.py with CBVs
  2. Update URL patterns to use .as_view()
  3. Verify templates still work (no changes needed for basic usage)
  4. Test all views: list, detail, create (logged in/out), update, delete
  5. Ensure only authors can edit/delete their posts

Best Practices & Summary

Best Practices:

  • Prefer generic CBVs for standard CRUD operations
  • Use mixins for shared behavior (e.g., LoginRequiredMixin)
  • Don’t overcomplicate—use FBVs for simple or highly custom logic
  • Always test after refactoring!

Summary:

  • CBVs reduce boilerplate and improve code structure
  • Django provides powerful generic views for common tasks
  • Customization is flexible via method overriding
  • Refactoring to CBVs makes apps more maintainable

Final Thought:

  • CBVs aren't always the answer—but when they fit, they save significant time and effort
