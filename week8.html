QuerySets and Database Lookups

QuerySets are lazy database query representations that enable efficient, chainable database operations.

Field Lookups:

  • exact, iexact - case-sensitive/insensitive equality
  • contains, icontains - substring matching
  • in, range - value ranges
  • gt, gte, lt, lte - comparisons
  • startswith, endswith - string boundaries

QuerySet Examples:

# Get published posts from 2023
Post.objects.filter(published_date__year=2023, status='published')

# Posts with titles containing "Django" (case-insensitive)
Post.objects.filter(title__icontains='django')

# Posts by authors with emails ending in "@example.com"
Post.objects.filter(author__email__endswith='@example.com')

# Chaining queries
Post.objects.filter(status='published').exclude(title__startswith='Draft')

Key Insight:

  • QuerySets are lazy - queries execute only when evaluated
  • Use double underscores (__) for field lookups and relationships

Aggregation, Annotation & Raw SQL

Django ORM provides powerful functions for data summarization and complex queries.

Aggregation Functions:

from django.db.models import Count, Avg, Max, Min, Sum

# Total number of posts
Post.objects.count()

# Average post length
Post.objects.aggregate(Avg('word_count'))

# Posts per author
Post.objects.values('author').annotate(post_count=Count('id'))

# Longest post
Post.objects.aggregate(Max('word_count'))

Raw SQL (When Needed):

# Raw queries for complex operations
Post.objects.raw('SELECT * FROM blog_post WHERE word_count > %s', [1000])

# Direct database connection for non-ORM operations
from django.db import connection
with connection.cursor() as cursor:
    cursor.execute("UPDATE blog_post SET views = views + 1 WHERE id = %s", [post_id])

Best Practice:

  • Prefer ORM over raw SQL for security and maintainability
  • Use extra() or raw() only when ORM can't express the query

Custom Model Managers

Custom managers encapsulate common query patterns and provide a cleaner API for your models.

Creating Custom Managers:

# models.py
class PublishedManager(models.Manager):
    def get_queryset(self):
        return super().get_queryset().filter(status='published')

class Post(models.Model):
    title = models.CharField(max_length=200)
    status = models.CharField(max_length=10, choices=[('draft', 'Draft'), ('published', 'Published')])
    published_date = models.DateTimeField(null=True, blank=True)
    
    objects = models.Manager()  # Default manager
    published = PublishedManager()  # Custom manager

Using Custom Managers:

# Get all published posts
published_posts = Post.published.all()

# Chain with other QuerySet methods
recent_published = Post.published.filter(published_date__year=2023)

# Custom methods in managers
class PublishedManager(models.Manager):
    def recent(self, days=7):
        cutoff = timezone.now() - timedelta(days=days)
        return self.get_queryset().filter(published_date__gte=cutoff)

# Usage: Post.published.recent(30)
  • Custom managers improve code readability and reusability
  • Always keep the default objects manager
  • Use get_queryset() to modify the base QuerySet

Database Optimization

Optimize database queries to prevent N+1 problems and improve application performance.

select_related() - ForeignKey Optimization:

# Without optimization - N+1 queries
posts = Post.objects.all()
for post in posts:
    print(post.author.username)  # Separate query for each post

# With select_related - single query with JOIN
posts = Post.objects.select_related('author')
for post in posts:
    print(post.author.username)  # No additional query

prefetch_related() - Reverse Relations:

# For ManyToMany or reverse ForeignKey
authors = Author.objects.prefetch_related('posts')
for author in authors:
    for post in author.posts.all():  # No additional queries
        print(post.title)

# Prefetch with filtering
from django.db.models import Prefetch
authors = Author.objects.prefetch_related(
    Prefetch('posts', queryset=Post.objects.filter(status='published'))
)

Performance Tips:

  • Use Django Debug Toolbar to identify N+1 queries
  • select_related for forward ForeignKey/OneToOne
  • prefetch_related for reverse ForeignKey/ManyToMany

Lab: Categories & Tags Implementation

Requirements:

  • Add Category model with name and description
  • Add Tag model with name (ManyToMany to Post)
  • Update Post model to include category ForeignKey
  • Implement filtering by category and tag in views
  • Display category/tag navigation in templates

Model Implementation:

# models.py
class Category(models.Model):
    name = models.CharField(max_length=100, unique=True)
    description = models.TextField(blank=True)

class Tag(models.Model):
    name = models.CharField(max_length=50, unique=True)

class Post(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    category = models.ForeignKey(Category, on_delete=models.CASCADE)
    tags = models.ManyToManyField(Tag, blank=True)
    author = models.ForeignKey(User, on_delete=models.CASCADE)
    created_at = models.DateTimeField(auto_now_add=True)
# views.py
def post_list(request):
    posts = Post.objects.select_related('category', 'author').prefetch_related('tags')
    category_id = request.GET.get('category')
    tag_id = request.GET.get('tag')
    
    if category_id:
        posts = posts.filter(category_id=category_id)
    if tag_id:
        posts = posts.filter(tags__id=tag_id)
    
    categories = Category.objects.all()
    tags = Tag.objects.all()
    return render(request, 'blog/post_list.html', {
        'posts': posts,
        'categories': categories,
        'tags': tags
    })
